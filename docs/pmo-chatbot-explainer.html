Scopes

Personal, Team, and Org views separate work so users see the right level of information.

Projects

Chat

Artifacts

Decisions

Approvals

Architecture Map

This is the shortest version of how a click in the browser turns into saved data and an AI response.

1. Browser

React screens

The user clicks tabs, sends messages, previews artifacts, and creates projects or chats.

2. Router

TanStack Start

3. Server functions

Actions

Functions such as sendChatMessage and createScopedChat validate the request.

4. Cloudflare

D1, R2, AI, Email

D1 stores records, R2 stores file bytes, Workers AI writes chatbot replies, and Email handles invites.

5. UI refresh

React Query

The frontend invalidates stale queries so the page shows the new message, task, idea, or project.

Detailed Build Notes

Open each section for the more specific implementation details behind the required documentation topics.

What the app does

It is a PMO workspace for AI-assisted project operations. Users can chat with VertexAI, capture ideas, inspect artifacts, track decisions, manage approvals, and update tasks.

It is structured around Personal, Team, and Org scopes, with separate project and workspace chats.

Frontend

The frontend is React 19 with TanStack Router, Query, Form, and Table. The main screen is in src/routes/index.tsx.

The UI uses componentized cards, dialogs, tables, filters, side rails, chat composer state, and optimistic pending chat messages.

Backend or server logic

Server logic uses TanStack Start server functions in src/lib/pmo-data.ts, src/lib/team-workflow.ts, and src/lib/auth-workflow.ts.

The Cloudflare Worker entry point passes the runtime environment into TanStack Start from src/worker.ts.

Data and storage

D1 stores structured records: users, sessions, teams, projects, chats, chat messages, ideas, artifacts metadata, actions, and invites.

R2 is configured for artifact files. D1 stores each artifact's title, metadata, href, and R2 object key.

Deployment

The app can build with npm run build. It is configured for Codex Sites through .openai/hosting.json and for Cloudflare Workers through wrangler.jsonc.

Cloudflare bindings include DB, ARTIFACTS_BUCKET, AI, and EMAIL.

Access and authentication

Better Auth handles email/password sign-in, email verification, sessions, and admin roles.

New accounts are invite-gated. Admins can create invites, roles can be admin, user, or viewer, and edit actions check role before writing.

Current Limitations

These are the practical gaps between the prototype and a hardened production product.

Area Current prototype behavior Why it matters
Seeded and mixed data Some workspace data is built from in-memory seed objects while team, project, chat, auth, and message records use D1. Production should use one authoritative persistence model so records do not reset or diverge.
Artifacts Artifact metadata and file links exist, with R2 configured, but generated-file lifecycle is still mostly demo-shaped. A real tool needs upload, versioning, permissions, scanning, retention, and durable generated outputs.
Sharing The UI exposes share actions and scoped invites, but magic-link governance is not yet a full product-grade sharing system. Real sharing needs expiration, revocation, audit logs, per-file permissions, and access warnings.
AI grounding Workers AI receives a priority D1-backed workspace context header with workspace name, active project, project status, and project description before recent chat messages or retrieved artifact context. Team project scoped RAG can also stream cited responses from indexed project artifacts. Production should keep retrieval permissions, citations, and controls against data leakage consistent across every chat and artifact path.
Authorization depth Chat and team/project membership checks exist for key write paths. Every read and write path should enforce ownership and scope consistently on the server.
Operational readiness The repo has build, seed, deploy, and type-check scripts. Production still needs monitoring, backups, migrations discipline, secrets management, rate limits, and incident procedures.

What Must Change For A Real Multi-User Tool

The short version: make the server the source of truth, make permissions strict, and make AI output traceable.

Data model

Move all ideas, artifacts, decisions, approvals, tasks, activity, and scope settings into D1 tables with migrations, indexes, ownership fields, and audit history.

Permissions

Apply server-side authorization to every read and write. Separate admin, editor, viewer, team owner, project member, and external shared viewer permissions.

Artifact system

Use R2 for files, D1 for metadata, and add versioning, generated-output tracking, virus scanning or file validation, preview conversion, and retention rules.

AI grounding

Connect chat to approved project sources, retrieve only allowed documents, cite sources, log prompts and responses, and add review controls for risky outputs.

Collaboration

Add comments, mentions, notifications, simultaneous editing expectations, activity feeds, and conflict handling when multiple users update the same project.

Operations

Add environment-specific secrets, automated tests, CI, database backups, alerting, rate limiting, billing or usage controls, and clear admin runbooks.
